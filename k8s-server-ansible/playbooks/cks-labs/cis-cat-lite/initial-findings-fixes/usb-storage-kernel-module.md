# Potential threat: usb-storage kernel module available


Ubuntu ships a general-purpose kernel, and usb-storage is a normal loadable module included so the same kernel works across laptops, bare metal servers, rescue environments, installers, and VMs.
CIS warns us that this is a potential source of trouble due to USB-based malware working via network infiltration and thus threatening a networked environment. Disabling the module should decrease the physical attack surface

## Beware
Disabling the usb-storage module will prevent you from using any USB storage devices, so a normal USB flash drive or external USB disk will usually not work on that host.

#### What happens when you plug it in:

* The USB device is physically detected.
* The kernel sees that it is a mass-storage device.
* The system would normally auto-load usb_storage.
* Your blacklist/install-override policy prevents that driver from loading.
* Without that driver, the device does not become a usable block device like /dev/sdb.

#### So in practice:

* the stick may still appear as a USB device at a low level
* but it will not be usable as mounted storage
* you typically will not get a working disk device for it

## CIS warning does not mean the module is currently loaded

To check if it is - run

``
lsmod | grep usb_storage
``

If no output - it is not loaded

``
modinfo usb_storage
``

If output exists - the module exists and is available to load

``
grep -R "usb-storage" /etc/modprobe.d /lib/modprobe.d
``

no output meands no override policies prevent the module from being loaded 

# Fix

### Unload module

```
 modprobe -r usb-storage 2>/dev/null
 rmmod usb-storage 2>/dev/null 
```

##### Explanation
We are telling the kernel to remove this module from memory now, if possible. After that, the module is no longer active in the current boot. But nothing prevents it from being loaded again later by hardware detection, udev, or an explicit modprobe.
So we need to set policy for future loadings and this is what Disable module does

### Disable module
   
   
1. Override install usb-storage (stronger of the two) by creating a .conf file with install usb_storage /bin/false in the /etc/modprobe.d/ directory 


```
 cat <<'EOF' | sudo tee /etc/modprobe.d/60-usb-storage.conf
 install usb-storage /bin/false
EOF
```

##### Explanation
modprobe reads rules from **/etc/modprobe.d/*.conf** before deciding what to do. 
When you run **modprobe usb-storage**, it finds the kernel object file and inserts it. 
With **install usb-storage /bin/false** - modprobe is told: when someone asks for usb-storage, run false instead of performing the normal module insertion. So even an explicit modprobe usb-storage is intercepted and fails.

2. Blacklist the module
Prevents automatic loading through aliases or hardware detection.

```
    cat <<'EOF' | sudo tee /etc/modprobe.d/60-usb-storage.conf
    blacklist usb-storage
EOF
```

##### Explanation 

###### alias-based auto-loading

A device is usually not announced to userspace as "load driver usb-storage". It is announced as "here are my bus/type/vendor/class identifiers". That identifier string is the alias. For example:
``
usb:vXXXXpXXXXdXXXXdcXXdscXXdpXXic08isc06ip50inXX
``

- ic08 = interface class 8, mass storage
- isc06 = SCSI transparent command set
- ip50 = Bulk-Only Transport

The usb-storage module advertises that it supports aliases matching those patterns. 
How does it work? Kernel modules embed alias metadata. Example:

``
modinfo usb-storage
``

That will show alias: lines exported by the module. Those aliases are collected into modprobe's lookup database, typically generated by depmod, for example in files like: /lib/modules/$(uname -r)/modules.alias

So modprobe performs a lookup:

- input: modalias string from device event
- lookup source: module alias database
- output: matching module, such as usb-storage


When we run blacklist usb-storage, we are telling modprobe:

if alias resolution says usb-storage is the answer, do not auto-load that module.

###### Full flow when device is detected

- A device appears.
- The kernel creates a device object in sysfs.
- That device object exposes a modalias string.
- The kernel sends a uevent.
- udev receives the event and asks modprobe to resolve the alias.
- modprobe checks its alias database and finds which module claims to support that alias.
- That module is loaded, unless policy blocks it.

Blacklisting can not stop someone from explicitly running **modprobe usb-storage**, which is why we need the install override.
Install override covers manual loading, blacklisting covers auto-loading.

### Difference between unload and disable
Unloading is about the current state of the running kernel. Disabling is about future load attempts, the persistent policy for the future.
